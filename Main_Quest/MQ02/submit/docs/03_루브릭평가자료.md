## 1. 데이터 EDA와 데이터 전처리

### 1-1. 데이터셋 개요

- **출처**: Kaggle NordTank586x371 (드론 촬영 풍력 터빈 이미지)
- **전체**: 13,470장 (결함 2,995장 / 정상 10,475장)
- **클래스**: Dirt(오염, class 0) / Damage(손상, class 1)

### 1-2. EDA 주요 발견

```
# 클래스별 박스 수 분포
Dirt   boxes:   581개  (전체의 6.2%)
Damage boxes: 8,770개  (전체의 93.8%)
→ Dirt:Damage = 1:15 극단적 불균형
```

- **정상 이미지 비중**: 10,475 / 13,470 = **77.8%** → 활용 방식이 성능에 직접 영향
- **박스 크기**: Damage의 86%가 64×64픽셀 이하 소형 객체 (586×371 이미지 기준)
  - 소형 객체 특성 → confidence threshold를 낮춰야 탐지율 확보 가능

### 1-3. 전처리 결정 사항

| 처리 항목 | 방법 | 근거 |

## 2. 모델 선정

### 2-1. YOLO 선정 이유

| 요구사항 | YOLO의 적합성 |
|---|---|
| 실시간 탐지 | 단일 forward pass → 30fps 이상 추론 가능 |
| 소형 객체 탐지 | YOLOv8 multi-scale head (P3/P4/P5) → 작은 박스도 탐지 |
| 드론 이미지 적용 사례 | 풍력 터빈 드론 탐지 선행 연구 다수 (YOLO 계열 우세) |
| 학습 용이성 | ultralytics API — 단 10줄로 train/val/predict 완결 |
| 사전학습 가중치 | COCO pretrained → 소규모 데이터에서도 빠른 수렴 |

### 2-2. 다른 모델과 비교

| 모델 계열 | 특징 | 이 태스크에서의 한계 |
|---|---|---|
| **Faster R-CNN** (2-stage) | 정확도 높음, anchor 기반 | 추론 속도 느림(~5fps), 실시간 불가 |
| **RT-DETR** | DETR 실시간 버전 | YOLO 수준 속도, Transformer 정확도 → 차기 실험 후보 |
| **YOLOv8 (채택)** | anchor-free, 빠름, 소형 객체 강함 | — |
| **YOLO11** | YOLOv8 후속, 유사 성능 | 비교 실험에서 YOLOv8s 대비 동등 수준 |

### 2-3. YOLO 모델 크기 선정

YOLOv8n/s, YOLO11n/s 4종을 동일 조건(labeled_only, 20 epoch)에서 비교:

| 모델 | mAP50 | Recall | Precision |
|---|---|---|---|
| YOLOv8n | 0.794 | 0.747 | 0.818 |
| **YOLOv8s ← 선정** | **0.818** | 0.750 | **0.854** |
| YOLO11n | 0.788 | 0.741 | 0.812 |
| YOLO11s | 0.799 | 0.746 | 0.851 |

> YOLOv8s: mAP50 최고(0.818), 파라미터 11M — 속도·정확도 균형점. 드론 탑재 가능한 22MB.

## 3. 성능 향상을 위한 논리적 접근

### 접근 전략

```
베이스라인(full 데이터) 확인
  → 문제 진단 (클래스 불균형, Damage 다양성 부족)
  → 가설 설정 → 실험 → 결과 분석 → 다음 가설
```

### 단계별 가설과 결과

| 단계 | 가설 | 실험 | 결과 |
|---|---|---|---|
| 1 | Dirt 증강으로 불균형 완화 → 성능↑ | full + Dirt×8 | Recall 0.769→0.751 ↓, Damage AP50 0.688→0.684 ↓ — Damage 학습 희석 |
| 2 | 외부 데이터(blade) → Damage 다양성↑ | full + blade | 전 지표 동등 이하 — 도메인 불일치로 기여 없음 |
| 3 | 증강 + blade 조합 → 시너지 | full + Dirt×8 + blade | 전 지표 최저 — 두 변화가 노이즈로 작용 |
| 4 | conf 하향 → 소형 객체 탐지 포함 | conf 0.25 → 0.20 | mAP50 0.7016→0.7137 ↑ — 해당 구간 실제 TP 확인 |

### 핵심 발견

> 데이터를 더 추가하고 증강을 더 한다고 반드시 좋아지지 않는다.
> 도메인 정합성이 맞지 않는 데이터는 오히려 학습을 방해한다는 것을 실험으로 확인.
> → **베이스라인(full dataset)이 최종 모델**

## 6. 적절한 Metric 설정과 사용 근거

### 6-1. 평가 지표 우선순위

| 순위 | 지표 | 선정 근거 |
|---|---|---|
| **1위** | **Recall (재현율)** | 결함 미탐지(FN) = 블레이드 파손 → 설비 사고 직결. 오탐(FP)은 재확인으로 보완 가능 |
| **2위** | **Damage AP50** | 심각도 공식에서 Damage 가중치 10 (Dirt의 5배). mAP50은 Dirt/Damage 동등 평균이라 서비스 가치와 괴리 |
| **3위** | mAP50 | 전반적 탐지 성능 참고 지표 |

### 6-2. mAP50을 1순위로 쓰지 않는 이유

```
mAP50 = (Dirt AP50 + Damage AP50) / 2

문제: Dirt를 잘 탐지해도 mAP50이 높아짐
     → Damage를 놓쳐도 Dirt만 잘 잡으면 좋은 모델처럼 보임
     → 안전 도메인에서 잘못된 최적화 방향
```

### 6-3. 공정 평가 설계

```
최종 비교 대상 4개 실험(full / full_aug / full_blade / full_aug_blade)은
모두 full dataset 기반 → 동일한 val 2,694장 공유
→ train-val 중복 없는 공정 비교
```

### 6-4. 목표 수치 설정 근거

- **Recall 0.70+**: 결함 10개 중 7개 이상 탐지 — 드론 점검 주기(월 1회)를 고려할 때 최소 허용 미탐율
- **Damage AP50 0.58+**: 초기 베이스라인 실험의 Damage AP50 수준을 기준으로 유지 또는 향상

## 8. Metric 결과 분석

### 8-1. 최종 모델 상태 진단 (exp_full_small_ep50, full val 2,694장)

| 지표 | 값 | 해석 |
|---|---|---|
| Recall | 0.769 | 결함 100개 중 약 77개 탐지 — 목표(0.70+) 달성 |
| Precision | 0.710 | 탐지 중 71%가 진짜 결함 |
| mAP50 | 0.752 | 전반적 탐지 성능 양호 |
| mAP50-95 | 0.517 | bbox 위치 정밀도는 보통 |
| Damage Recall | 0.645 | Damage 100개 중 약 65개 탐지 |
| Damage AP50 | 0.688 | Damage 단독 탐지 성능, 목표(0.58+) 달성 |

### 8-2. Damage Recall이 전체 Recall보다 낮은 이유

전체 Recall 0.769 대비 Damage Recall은 0.645다. Damage의 86%가 64×64픽셀 이하 소형 객체라 탐지 난이도가 높기 때문이다. 이를 보완하기 위해 inference conf를 0.20으로 낮춰 저신뢰도 구간의 실제 탐지(TP)를 포함시켰고, 서비스 레벨에서는 confidence 구간 플래그(0.4~0.7 = 재확인 필요)로 오탐 위험을 관리한다.

### 8-3. mAP50과 mAP50-95의 괴리

mAP50=0.752 대비 mAP50-95=0.517로 괴리가 있다. 이는 bbox의 존재와 대략적 위치는 잡지만 정밀한 경계 예측은 상대적으로 약하다는 의미다.

원인은 데이터 특성에 있다. Damage의 86%가 64×64픽셀 이하 소형 객체(586×371 이미지 기준)라서, 작은 박스를 IoU 0.95 수준으로 정밀하게 맞추는 것 자체가 어렵다. bbox 정밀도보다 탐지 여부가 중요한 도메인이므로 conf 조정으로 탐지율을 확보하는 방향을 우선했다.

### 8-4. conf=0.20 분석

```
model.val()은 threshold를 내부 sweep → conf 변경 시 Recall 불변
대신 mAP50 변화로 실제 TP 존재 확인 가능

conf 0.25 → 0.20:
  mAP50        0.7016 → 0.7137  (+0.0121)
  Damage AP50  0.6045 → 0.6287  (+0.0242)

→ conf 0.20~0.25 구간에 실제 결함 탐지가 존재
→ predict() 추론에서 conf=0.20 적용 시 이 구간의 탐지 포함
```

### 8-5. 결과 해석 요약

본 프로젝트의 결과 분석 우선순위:

1. **Recall** — 미탐율 확인. 0.769로 목표 달성
2. **Damage AP50** — Damage 단독 탐지 성능. 0.688로 목표 달성
3. **mAP50** — 전반적 탐지 성능 참고
4. **mAP50-95** — 소형 객체 특성상 낮게 나오는 것은 예상된 결과. bbox 정밀도 개선은 해상도·모델 크기 조정으로 접근 (차기 과제)

## 9. 배포 설계 (서비스 연계)

### 심각도 스코어링

```python
# 객체 점수 = 클래스가중치 × 크기가중치 × confidence
# 클래스가중치: Damage=10, Dirt=2
# 크기가중치: 1 + (박스면적비율 × 5)
터빈_위험도 = Σ(각 탐지 객체 점수)
```

### 등급별 에스컬레이션

| 점수 | 등급 | 알림 채널 | 대상 |
|---|---|---|---|
| 0 | 정상 | 로그만 | — |
| 1~15 | 관찰 | 대시보드 | 현장 담당자 |
| 15~40 | 주의 | 대시보드 + 이메일 | 담당자 + 팀 |
| 40+ | 위험 | 대시보드 + Slack 즉시 경보 | 담당자 + 관리자 |

### conf threshold 전략

```
inference conf = 0.20  (소형 객체 탐지 포함)

서비스 레벨 처리:
≥ 0.7   → 자동 경보 (확실한 탐지)
0.4~0.7 → "재확인 필요" 플래그 (human-in-the-loop)
< 0.4   → 무시 (오탐 가능성)
```
