# 나의 터빈일지 — 데이터 분석 & 모델 선정 파트 정리 (발표자료용)

> 작성: 채진현 (데이터 분석 / 전처리 / 모델 선정 담당)
> 용도: 발표자료 정리를 위해 전달 — 분석 흐름 + 의사결정 근거 정리

## 1. 프로젝트 진행 흐름 (팀 협업)

```
[1] 전체 아이디어 회의 (팀 전원)
      · 풍력 터빈 손상 탐지를 "어떤 서비스로 만들까" 방향 논의
              ↓
[2] 데이터 분석 & 모델 선정 (채진현)
      · EDA → 평가 지표 설계 → 전처리 결정 → 모델 비교 → 최적화
      · 팀 회의/피드백을 반영해 실험 방향 조정
              ↓
[3] 서비스 상세 설계 및 구현 (강경수)
      · 분석 결과(지표 설계, 등급 기준, 가중치)를 서비스에 이식
```

핵심: 아이디어(전원) → 데이터·모델(채진현) → 서비스 구현(강경수)으로 이어지는 흐름.
분석 단계에서 정의한 **Damage 우선 전략과 4등급 에스컬레이션 설계**가 서비스 전체의 뼈대가 됐다.

## 3. 분석 흐름 & 주요 발견

### 3-1. EDA

**데이터 구성**

| 구분 | 수치 |
| --- | --- |
| 전체 이미지 | 13,470장 (해상도 586×371) |
| 결함 이미지 | 2,995장 (22.2%) |
| 정상 이미지 | 10,475장 (77.8%) |
| Dirt 박스 (train) | 468개 (6.3%) |
| Damage 박스 (train) | 6,993개 (93.7%) |

**중요 발견 두 가지**

① **Dirt : Damage = 1 : 15 극단적 불균형**
→ mAP50이 Dirt와 Damage를 동등 평균하므로 왜곡됨 → Damage 클래스만 별도 추적 필요

② **Damage의 86%가 64×64px 이하 소형 객체** (너비·높이 모두 기준, 7,513/8,770개)
→ multi-scale detection head 필수 / confidence threshold를 낮춰 저신뢰도 탐지 포함 전략 수립

### 3-2. 외부 데이터 중복 검증

Roboflow에서 확보한 3종 데이터셋을 **코드로 직접 해시 비교**해 중복 여부를 확인했다.

| 데이터셋 | 결과 |
| --- | --- |
| Wind Turbine v1i | NordTank와 **100% 중복** → 제외 |
| Wind Turbine v11i (DTU) | NordTank와 **100% 중복** → 제외 |
| Wind Turbine Blade v7i | **완전 신규** → 실험 편입 |

→ 중복 확인 없이 썼다면 학습 데이터에 val 데이터가 섞이는 데이터 누수가 발생할 뻔했다.

## 4. 평가 지표 설계 — 왜 Recall을 1순위로 잡았는가

```
FN (미탐지):  결함 → 정상 판정 → 블레이드 파손 방치 → 수억원 손실 / 안전사고
FP (오탐지):  정상 → 결함 판정 → 재확인 작업 발생 → human-in-the-loop로 관리 가능

결론: FP 비용 << FN 비용  →  Recall 극대화 전략 채택
```

**지표 우선순위**

| 순위 | 지표 | 이유 |
| --- | --- | --- |
| 1 | **Recall** | FN = 결함 미탐지 = 사고 직결 |
| 2 | **Damage AP50** | 서비스 Damage 가중치=10 (Dirt의 5배)와 직결 |
| 참고 | mAP50 | Dirt=Damage 동등 평균 → 불균형 상황에서 왜곡 |

→ 이 지표 설계가 강경수의 서비스 위험도 공식(Damage 가중치 10, Dirt 가중치 2)으로 그대로 이어졌다.

## 5. 데이터셋 전처리 결정

### 5-1. 정상 이미지를 전체 포함한 이유

정상 이미지(10,475장)를 버리면 모델이 정상 블레이드를 한 번도 보지 못한다.
실제 운영 환경에서 정상 이미지가 대부분인데, 정상 표면을 결함으로 오탐하는 FP가 폭증한다.

YOLO는 빈 라벨(.txt)을 가진 이미지를 **background image**로 학습에 활용한다.
정상 이미지를 background image로 포함 → "결함이 아닌 것"의 경계를 학습 → FP 억제, Precision 안정화

결과: full dataset이 4종 비교에서 Recall·Damage AP50 최고, Precision도 가장 높았다.

### 5-2. 정상 이미지 비율 조정 실험을 제외한 이유 (val set 공정성)

정상 이미지 일부만 포함하거나 비율을 조정하는 실험도 고려했다.
하지만 이를 적용하면 **train과 val에서 정상 이미지 비율이 달라진다**.

4개 full 변형 실험이 모두 동일한 val 2,694장을 공유하도록 설계했기 때문에,
val 구성을 건드리는 실험은 비교 기준 자체를 무너뜨린다.
→ "더 쉬운 val을 쓴 것"인지 "진짜 더 잘하는 것"인지 구별이 불가능해진다.

따라서 **train 구성만 다르고 val은 고정**하는 방식으로 공정 비교를 보장했다.
정상 이미지 비율 조정은 이 원칙에 맞지 않아 실험 범위에서 제외했다.

### 5-3. Dirt×8 증강이 역효과인 이유

Dirt:Damage = 1:15 불균형 해소를 위해 Dirt 이미지를 8배 증강했지만:

| 구성 | Recall | Damage AP50 | Precision |
| --- | --- | --- | --- |
| full baseline | **0.769** | **0.688** | 0.710 |
| + Dirt×8 | 0.751 ↓ | 0.684 ↓ | 0.674 ↓ |

Dirt를 늘리자 학습 배치 내 Dirt 비중이 높아져 **Damage 학습 신호가 희석됐다**.
클래스 불균형을 해소하려다 반대 방향으로 불균형을 만든 셈.
→ 미적용. 불균형은 서비스 단 가중치(Damage=10)로 보정.

### 5-4. blade 외부 데이터가 역효과인 이유

| 항목 | NordTank | Blade v7i |
| --- | --- | --- |
| 해상도 | 586×371 | 384×384 |
| 터빈 기종 | 수평축 터빈 | 다른 기종 |
| 결함 패턴 | Dirt + Damage | crack(→Damage 리맵) |

도메인 불일치 → 학습 노이즈로 작용 → Recall 0.769→0.763 하락
→ 미적용. 같은 도메인 추가 데이터가 없다면 외부 데이터 편입은 효과 없다는 결론.

## 6. 모델 선정 — 왜 YOLOv8s인가

### 6-1. 왜 YOLO 계열인가

| 요구사항 | 근거 |
| --- | --- |
| 드론 탑재 가능 | 단일 forward pass → 경량·실시간 탐지 |
| 소형 객체 탐지 | P3/P4/P5 multi-scale head |
| 소규모 데이터 | COCO 사전학습 가중치 → 빠른 수렴 |

### 6-2. 4종 비교 결과 (labeled-only, 20 epoch, val 599장)

| 모델 | mAP50 | Recall | Precision | 크기 |
| --- | --- | --- | --- | --- |
| YOLOv8n | 0.794 | 0.747 | 0.818 | 6MB |
| **YOLOv8s ✅** | **0.818** | **0.750** | **0.854** | **22MB** |
| YOLO11n | 0.788 | 0.741 | 0.812 | 5.4MB |
| YOLO11s | 0.799 | 0.746 | 0.851 | 18MB |

**YOLOv8s 선정 이유**
- mAP50 최고(0.818) + Recall 최고(0.750) + Precision 최고(0.854)
- 22MB — 드론 탑재 가능한 경량 경계점
- YOLO11s 대비 전 지표 소폭 우세 → Recall 1순위 기준에서 YOLOv8s 승

## 7. conf 최적화 — 왜 0.20인가

**val() vs predict() 동작 이해가 핵심이었다**

| 함수 | conf 반영 | Recall 변화 | 용도 |
| --- | --- | --- | --- |
| `model.val()` | 내부 threshold sweep | 변화 없음 | 평가 지표 산출 |
| `model.predict()` | conf 파라미터 직접 적용 | conf 낮출수록 증가 | 실제 추론 |

→ val()에서 conf를 낮춰도 Recall이 바뀌지 않는다. 대신 **mAP50이 올라가면 그 구간에 실제 TP가 존재**한다는 증거다.

| conf | mAP50 | Damage AP50 |
| --- | --- | --- |
| 0.25 (기본값) | 0.7016 | 0.6045 |
| **0.20 ✅** | **0.7137** | **0.6287** |

→ mAP50 +0.012, Damage AP50 +0.024: conf 0.20~0.25 사이에 실제 손상 탐지(TP)가 있었다.
Damage의 86%가 소형 객체 → 낮은 confidence가 나오는 게 당연한 구조.
conf=0.20으로 이 구간을 포함하면 더 많은 실제 결함을 잡을 수 있다.

## 9. 기억에 남는 순간들

**① "데이터를 더 늘리면 무조건 좋아진다"는 직관이 틀렸을 때**

Dirt×8 증강을 넣었더니 Recall이 0.769 → 0.751로 떨어졌다.
불균형을 해소하려고 한 조작이 오히려 Damage 학습을 방해한 것.
데이터 양보다 **학습 배치에서 신호가 어떻게 구성되느냐**가 더 중요하다는 걸 실험으로 직접 확인했다.

**② val()과 predict()의 동작 차이를 이해하기까지**

처음에는 "conf를 낮추면 Recall이 올라간다"고 단순하게 생각했다.
그런데 val()로 평가하면 conf를 어떻게 줘도 Recall이 바뀌지 않았다.
원인을 파고들었더니 val()은 threshold를 0~1 전체를 sweep해서 PR 곡선을 그린 뒤 AP를 계산하는 구조였다.
mAP50의 변화가 오히려 "이 구간에 실제 TP가 있다"는 증거라는 걸 깨닫고 conf=0.20을 확정했다.

**③ 외부 데이터 중복 확인 — 안 했으면 데이터 누수였을 뻔**

Roboflow에서 찾은 데이터 2종이 NordTank와 100% 겹치는 것을 해시 비교로 확인했다.
만약 그냥 합쳐 썼다면 val 이미지가 train에 포함되는 데이터 누수가 발생했을 것.
"유명한 데이터셋이면 다 다를 것"이라는 가정 없이 **직접 검증**하는 습관이 중요하다.

**④ 베이스라인이 최고였을 때**

Dirt 증강도 해보고, 외부 데이터도 넣어보고, 둘 다 합쳐봤는데 전부 베이스라인보다 낮거나 같았다.
아무것도 안 한 게 가장 좋은 결과. 처음엔 "내가 뭔가 잘못한 건가" 싶었는데,
결국 NordTank 도메인에 완전히 맞는 데이터만 있었기 때문에 **불순물 없는 원본이 가장 강했다**.

## 10. 발표에서 강조하면 좋은 포인트

1. **지표 설계가 서비스 설계로 이어짐** — Recall 1순위 결정이 위험도 가중치(Damage=10) 직결
2. **데이터 검증을 코드로 직접 했음** — 중복 검증, 라벨 파일 분석 (수치 신뢰 근거)
3. **증강·외부 데이터가 역효과였던 실험** — "더하면 좋다"는 가정을 실험으로 반증
4. **val() 동작 이해로 conf 최적화** — 단순 파라미터 튜닝이 아닌 모델 동작 원리 이해에서 출발
5. **val set 고정으로 공정 비교** — 실험 설계의 신뢰성 확보
