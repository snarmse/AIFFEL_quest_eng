# 나의 터빈일지 — 서비스 데모 설계서
### AI 손상 탐지 기반 관리자 알람 시스템

> 단순 탐지에서 그치지 않고, **현장 관리자가 실제로 쓸 수 있는 점검·알람 도구**로 발전.
> 최종 모델(YOLOv8s, `exp_full_small_ep50/best.pt` + conf=0.20, Recall 0.769 / Damage AP50 0.688)의 탐지 결과 위에 "심각도 판정 → 등급 → 에스컬레이션 알람"을 얹는다.

---

## 1. 왜 이 방향인가

풍력 터빈은 해상·산간에 있어 사람이 매번 점검하기 어렵다.
드론이 찍은 이미지를 AI가 분석해 **손상이 감지될 때만, 중요도에 맞는 방식으로** 관리자에게 알린다.
관리자는 모든 터빈을 볼 필요 없이 **경보가 온 것만** 확인하면 된다.

이는 AI 비전 탐지의 **검증된 B2B 모델**(예방 정비, predictive maintenance)이다.
새로운 기능을 발명하기보다, **"당연한 흐름을 완성도 높게"** 만드는 것이 이 도메인의 핵심이다.

---

## 2. 전체 흐름

```
이미지/영상 입력
  → YOLO 탐지 (Dirt / Damage)
  → 심각도 점수 계산 (클래스 × 크기 × 신뢰도)
  → 등급 판정 (정상 / 관찰 / 주의 / 위험)
  → 등급별 에스컬레이션 (채널·대상 차등)
  → 점검 이력에 기록
```

---

## 3. 심각도 스코어링

### 공식
```
터빈 위험도 = Σ(각 탐지 객체 점수)
객체 점수 = 클래스가중치 × 크기가중치 × 신뢰도

- 클래스가중치:  Damage = 10,  Dirt = 2   (손상이 오염보다 5배 위험)
- 크기가중치:    1 + (박스 면적비율 × 5)    (큰 손상일수록 최대 6배)
- 신뢰도:        confidence (0~1)
```

### 설계 근거
- **손상 > 오염**: 손상은 사고 직결, 오염은 청소로 해결 → 클래스 가중치 차등
- **클수록 위험**: 큰 파손이 작은 균열보다 위험 → 박스 면적 반영
- **모델 확신도**: 확실한 탐지에 더 무게 → confidence 곱

---

## 4. 등급 분류

| 점수 | 등급 | 의미 |
|---|---|---|
| 0 | 🟢 정상 | 이상 없음 |
| 1~15 | 🟡 관찰 | 다음 정기점검 시 확인 |
| 15~40 | 🟠 주의 | 점검 계획 수립 |
| 40+ | 🔴 위험 | 즉시 점검 필요 |

> 임계치는 실제 데이터로 조정. (손상 이미지를 넣어보며 등급 구간을 튜닝)

---

## 5. ★ 에스컬레이션 정책 (핵심 차별점)

실제 운영에서 **모든 이상을 똑같이 알리면 사람들이 알림을 무시하기 시작한다.**
그래서 **중요도에 비례해, 채널 강도와 대상 범위를 함께 차등**한다.
(현장 운영 경험 기반 — 이는 단순 알림이 아니라 **의사결정·책임 흐름**이다.)

| 등급 | 알림 방법 (채널 강도) | 알림 대상 (범위) |
|---|---|---|
| 🟢 정상 | 로그만 기록 | - |
| 🟡 관찰 | 대시보드 표시 | 현장 담당자 |
| 🟠 주의 | 대시보드 + 이메일/팀 메시지 | 담당자 + 팀 |
| 🔴 위험 | 대시보드 + 즉시 Slack/SMS 경보 | 담당자 + **관리자/책임자** |

**두 축으로 차등:**
- **채널 강도**: 로그 → UI → 이메일 → 즉시 경보 (위험할수록 push 강도 ↑)
- **대상 범위**: 담당자 → 팀 → 관리자 (위험할수록 상급자까지 에스컬레이션)

**왜 이렇게?**
- 오염 하나에 책임자까지 알림 가면 → 아무도 안 본다 (알림 피로)
- 심각한 손상이 로그에만 있으면 → 놓쳐서 사고
- 중요도에 비례하는 알림이라야 "🔴 알림 = 진짜 중요" 라는 신뢰가 유지된다

---

## 6. 오탐 관리 (모델 한계를 제품 설계로 보완)

최종 모델에서 **Damage 탐지가 Dirt 대비 어려움**(Damage Recall 0.645 — 소형 객체 특성)을 확인했다.
이를 숨기지 않고, **애매한 탐지는 자동 판정하지 않고 사람이 확인**하도록 설계한다.

```
- confidence ≥ 0.7  → 정상 반영, 확실한 경보
- 0.4 ~ 0.7        → "⚠️ 재확인 필요" 플래그 (담당자 눈으로 확인)
- < 0.4            → 무시 (오탐 가능성)
```

특히 🔴 위험이라도 confidence가 애매하면 **자동 경보 대신 담당자 재확인 요청**으로 한 단계 낮춘다.
→ 오탐으로 책임자를 불필요하게 호출하는 것을 방지. (실제 서비스의 신뢰를 지키는 핵심)

---

## 7. 구현 범위 (데모 현실화)

실제로 SMS·이메일을 다 연결하지 않아도, **"이 등급이면 이 채널로 이 사람에게 간다"를 보여주는 것**으로 충분하다.

- **UI 대시보드** (Streamlit): 업로드 → 탐지 결과 + 박스 + 통계 → 등급 + 알림
- **Slack 웹훅**: 1개만 실제 연결 (🔴 위험 시 실제로 채널에 발송 → 시연 임팩트)
- **SMS/이메일**: "발송됨" 표시로 시뮬레이션 (누구에게 갔는지 화면에 표기)
- **점검 이력**: 감지 내역을 로그 테이블로 기록 ("터빈일지" 컨셉과 연결)

---

## 8. 발표 시연 시나리오

등급별 대응이 **눈에 보이게** 3단계로 시연:

```
1. 정상 이미지 입력   → 🟢 "정상, 로그만 기록" (조용함)
2. 오염 이미지 입력   → 🟡/🟠 "대시보드 표시 + 팀 알림"
3. 심각한 손상 입력   → 🔴 "위험! Slack 경보 + 관리자 호출"
                        (실제 Slack에 알림 뜨는 것 시연)
```

**발표 메시지:**
> "단순히 손상을 찾는 데 그치지 않고, 현장 관리자가 바로 쓸 수 있는 도구로 만들었습니다.
> 실제 운영에서는 모든 알림을 똑같이 보내면 무시당하기 때문에, **중요도에 따라 채널과 대상을 나눴습니다.**
> 또한 모델의 Damage 탐지 한계를 확인했기에, **애매한 건 자동 경보 대신 사람이 재확인**하도록 설계했습니다."

| 5 | 발표 시연 준비 (정상/오염/손상 3케이스) | 0.5일 |
