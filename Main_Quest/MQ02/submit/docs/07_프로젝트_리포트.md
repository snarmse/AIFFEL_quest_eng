# 풍력 터빈 블레이드 결함 탐지 프로젝트 리포트

> 해커톤 프로젝트 — 데이터 분석부터 모델 선정까지

---

## 1. 프로젝트 배경 및 목적

### 문제 정의

풍력 터빈 블레이드는 고속 회전과 외부 충격으로 인해 **Dirt(오염)** 및 **Damage(균열·손상)** 이 발생하기 쉽다. 육안 점검을 위해 작업자가 수십 미터 높이에 올라가야 하는 기존 방식은 비용과 안전 위험이 크다.

**목표**: 드론 촬영 이미지에서 실시간으로 결함을 탐지하고, 심각도에 따라 관리자에게 자동 에스컬레이션 알림을 보내는 AI 파이프라인 구축.

### 핵심 전제

| 상황 | 결과 | 비용 |
|---|---|---|
| FN (미탐지): 결함 → 정상 판정 | 블레이드 파손 방치 → 수억원 손실 / 안전사고 | **복구 불가** |
| FP (오탐지): 정상 → 결함 판정 | 재확인 작업 발생 | 인력·시간 비용 (관리 가능) |

→ **FP 비용 << FN 비용** → Recall 극대화 전략 채택

## 2. 데이터셋 EDA

### 2-1. 기본 데이터셋

- **출처**: Kaggle NordTank586x371 (드론 촬영 수평축 풍력 터빈)
- **전체 이미지**: 13,470장 / 해상도: 586×371 px
- **클래스**: Dirt(오염, class 0) / Damage(손상, class 1)

| 구분 | 수량 |
|---|---|
| 결함 이미지 (라벨 있음) | 2,995장 (22.2%) |
| 정상 이미지 (라벨 없음) | 10,475장 (77.8%) |

### 2-2. 클래스 불균형 (train split 기준)

| 클래스 | BBox 수 | 비율 |
|---|---|---|
| Dirt (class 0) | 468개 | 6.3% |
| Damage (class 1) | 6,993개 | 93.7% |
| **합계** | **7,461개** | — |

→ **Dirt : Damage = 1 : 14.9** (극단적 불균형)  
→ 단순 학습 시 Dirt 미탐지 위험 / mAP50이 왜곡될 수 있음

### 2-3. 소형 객체 문제

BBox 크기 분포 (너비 기준, train+val 9,351개 전체):

| 구간 | BBox 수 | 비율 |
|---|---|---|
| < 32px | 6,305개 | 67.4% |
| 32–64px | 1,555개 | 16.6% |
| 64–128px | 811개 | 8.7% |
| ≥ 128px | 680개 | 7.3% |
| **64px 이하 합계** | **7,860개** | **84.0%** |

- **Damage만 기준 (너비·높이 모두 64px 이하)**: 7,513 / 8,770 = **85.7% (≈86%)**
- 중앙값: 너비 19px, 높이 20px

→ 객체가 극히 소형 → multi-scale detection head가 필수 / confidence threshold 전략이 중요

### 2-4. 외부 데이터 검증

추가 확보한 외부 데이터 3종을 직접 중복 검증했다.

| 데이터셋 | 이미지 수 | 검증 결과 | 처리 |
|---|---|---|---|
| Wind Turbine v1i (Roboflow) | 2,191장 | NordTank와 **100% 중복** | ❌ 미사용 |
| Wind Turbine v11i (Roboflow DTU) | 2,979장 | NordTank와 **100% 중복** | ❌ 미사용 |
| Wind Turbine Blade v7i | 2,092장 | **완전 신규** (384×384, 다른 터빈 기종) | ✅ 비교 실험 편입 |

- Blade v7i: 1개 클래스(crack) → Damage(class 1)로 전체 리맵핑
- valid/test 618장은 도메인 혼재 우려로 제외 → train 1,474장만 실험에 사용

## 3. 평가 지표 설계

### 지표 우선순위

| 순위 | 지표 | 선정 이유 |
|---|---|---|
| **1** | **Recall** | FN = 결함 미탐지 = 사고 직결. FP는 human-in-the-loop로 관리 가능 |
| **2** | **Damage AP50** | 서비스 가중치 Damage=10 (Dirt의 5배). 서비스 가치와 직결 |
| 참고 | mAP50 | Dirt=Damage 동등 평균 → 클래스 불균형 상황에서 왜곡 가능성 |

### 목표값 설정

| 지표 | 목표 | 근거 |
|---|---|---|
| Recall | 0.70 이상 | 결함 30% 이상 미탐지 시 서비스 신뢰 붕괴 |
| Damage AP50 | 0.58 이상 | Damage 탐지율 일정 수준 이상 보장 |

### mAP50을 주 지표로 삼지 않은 이유

mAP50 = (Dirt AP50 + Damage AP50) / 2  
→ Dirt:Damage = 1:15 불균형에서 Dirt의 낮은 탐지 성능이 평균을 끌어내림  
→ 서비스 관점에서는 Damage 탐지가 훨씬 중요하므로 Damage AP50을 별도 추적

## 4. 모델 선정

### 4-1. YOLO를 선택한 이유

| 요구사항 | 이유 |
|---|---|
| 드론 탑재 | 경량 모델 필요 (엣지 디바이스) |
| 실시간 탐지 | 단일 forward pass (two-stage보다 빠름) |
| 소형 객체 | multi-scale detection head (P3/P4/P5) |
| 앵커 없이 유연한 탐지 | anchor-free 구조 (YOLOv8 이후) |

### 4-2. 4종 모델 비교 실험

**조건**: labeled_only 데이터셋, 20 epoch, labeled val 599장

| 모델 | mAP50 | mAP50-95 | Recall | Precision | 파라미터 |
|---|---|---|---|---|---|
| YOLOv8n | 0.794 | 0.553 | 0.747 | 0.818 | 3.2M / 6MB |
| **YOLOv8s ✅** | **0.818** | **0.575** | **0.750** | **0.854** | 11M / 22MB |
| YOLO11n | 0.788 | 0.534 | 0.741 | 0.812 | 2.6M / 5.4MB |
| YOLO11s | 0.799 | 0.559 | 0.746 | 0.851 | 9.4M / 18MB |

### 4-3. YOLOv8s 선정 이유

- **mAP50 최고 (0.818)** + **Precision 최고 (0.854)**
- 22MB — 드론 탑재 가능한 경량 경계점
- nano 대비 mAP50 +2.4%p, YOLO11s 대비 mAP50 +1.9%p
- YOLO11s와 근소하지만 전 지표에서 우세 → YOLOv8s 채택

## 5. 데이터셋 구성 실험

### 5-1. full dataset을 선택한 이유

labeled_only (결함 이미지만 학습) vs full dataset (정상 이미지 포함):

- labeled_only 학습 → 모델이 정상 이미지를 한 번도 보지 못함
- 실제 추론 시 정상 터빈 이미지에서 **FP 폭증** 발생
- 정상 이미지를 빈 라벨(.txt)로 background image 처리 → YOLO의 FP 억제 학습 메커니즘 활용
- **full dataset이 Precision 안정화에 직결**

### 5-2. 5종 데이터셋 구성

| 데이터셋 | Train | Val | 특징 |
|---|---|---|---|
| labeled_only | 2,396 | 599 | 결함 이미지만 — 모델 선정 실험 전용 |
| **full (baseline) ✅** | **10,776** | **2,694** | 전체 포함 — 최종 채택 |
| full + Dirt×8 | 14,400 | 2,694 | Dirt 이미지 8배 증강 |
| full + blade | 12,250 | 2,694 | blade 외부 데이터 1,474장 추가 |
| full + Dirt×8 + blade | 15,874 | 2,694 | 증강 + 외부 데이터 모두 적용 |

> val 2,694장은 4개 full 변형 모두 동일 → 공정 비교 보장

### 5-3. 비교 실험 결과 (YOLOv8s, 50 epoch, val 2,694장)

| 데이터셋 | Precision | Recall | mAP50 | mAP50-95 | Damage AP50 |
|---|---|---|---|---|---|
| **Baseline (full) ✅** | **0.710** | **0.769** | **0.752** | 0.517 | **0.688** |
| + Dirt×8 증강 | 0.674 | 0.751 | 0.718 | 0.515 | 0.684 |
| + Blade 외부 데이터 | 0.710 | 0.763 | 0.748 | 0.517 | 0.688 |
| + Dirt×8 + Blade | 0.645 | 0.735 | 0.726 | **0.525** | 0.682 |

### 5-4. 실험별 분석

**Dirt×8 증강이 역효과인 이유**

Dirt 클래스 1:15 불균형 완화를 위해 Dirt 이미지를 8배 증강 (468 → 4,648개)했으나:
- 학습 배치에서 Dirt가 과도하게 등장 → **Damage 탐지 학습 신호 희석**
- Recall 0.769 → 0.751 하락, Damage AP50 0.688 → 0.684 하락
- 클래스 불균형은 class_weight (Damage=10, Dirt=2)로 별도 보정

**blade 외부 데이터가 역효과인 이유**

- NordTank: 586×371px, 수평축 터빈
- Blade v7i: 384×384px, 다른 기종의 터빈 (다른 결함 패턴)
- **도메인 불일치** → 학습 노이즈로 작용
- Recall 0.769 → 0.763 하락, Damage AP50 동일 (기여 없음)

→ **결론: 원본 full dataset (증강·외부 데이터 없음)이 전 지표 최고**

## 6. conf 최적화

### 6-1. val() vs predict() 동작 이해

| 함수 | conf 반영 방식 | Recall 변화 | 용도 |
|---|---|---|---|
| `model.val()` | 내부 임계값 스윕 (0~1 전체) | conf 파라미터와 무관 | 평가 지표 산출 |
| `model.predict()` | conf 파라미터 직접 적용 | conf 낮출수록 증가 | 실제 추론 |

핵심 인사이트: **val()로 mAP50을 계산할 때 conf가 달라도 Recall은 변하지 않는다.**  
하지만 conf를 낮추면 실제 TP가 늘어나므로 mAP50 자체가 올라갈 수 있다.

### 6-2. conf 비교 결과

| conf | mAP50 | Damage AP50 | Recall | Damage Recall |
|---|---|---|---|---|
| 0.25 (기본값) | 0.7016 | 0.6045 | 0.7691 | 0.6449 |
| **0.20 ✅** | **0.7137** | **0.6287** | **0.7691** | **0.6449** |
| 변화량 | **+0.0121** | **+0.0242** | 동일 | 동일 |

- mAP50 상승: conf 0.20~0.25 구간에 **실제 TP가 존재**한다는 증거
- 소형 객체 (중앙값 19×20px)는 낮은 confidence로 탐지됨
- **conf=0.20 채택**: 소형 객체 탐지율 향상 + mAP50 개선

## 8. 서비스 설계

### 8-1. 심각도 스코어링

```
터빈 위험도 = Σ(클래스가중치 × 크기가중치 × confidence)

클래스가중치 : Damage=10, Dirt=2
크기가중치   : 1 + (BBox 면적비율 × 5)
```

Damage를 Dirt보다 5배 가중 — 서비스 피해 비용 차이 반영

### 8-2. 등급 분류 및 에스컬레이션

| 점수 | 등급 | 알림 |
|---|---|---|
| 0 | 🟢 정상 | 로그만 기록 |
| 1~15 | 🟡 관찰 | 대시보드 표시 |
| 15~40 | 🟠 주의 | 대시보드 + 이메일 |
| 40+ | 🔴 위험 | Slack 즉시 경보 |

### 8-3. Human-in-the-Loop

| confidence | 처리 방식 |
|---|---|
| ≥ 0.7 | 자동 경보 처리 |
| 0.4 ~ 0.7 | ⚠️ 재확인 필요 플래그 |
| < 0.4 | 무시 (노이즈 제거) |

오탐(FP) 관리: confidence가 애매한 구간은 자동 처리하지 않고 사람이 최종 확인하도록 설계  
→ Recall 우선 전략(낮은 conf)과 오탐 부담 사이의 균형점

## 10. 회고

### 잘 된 점

- **외부 데이터 직접 검증**: 두 Roboflow 데이터셋이 100% 중복임을 코드로 확인해 불필요한 실험 제거
- **val() 동작 이해**: threshold sweep과 conf 파라미터의 차이를 정확히 이해해 conf 최적화 방향 수립
- **공정 비교 설계**: 4개 full 변형 모두 동일 val 2,694장 사용 → 실험 간 비교 신뢰성 확보
- **지표 설계 명확화**: mAP50 대신 Recall·Damage AP50 우선순위화 → 서비스 목적에 맞는 판단 가능

### 어려웠던 점

- 소형 객체 탐지 한계: 19×20px 중앙값 객체는 어떤 모델도 완벽히 탐지하기 어려움
- Dirt×8 증강 역효과: 불균형 해소를 위한 전통적 접근이 오히려 Damage 학습을 방해
- 도메인 불일치 문제: 외부 blade 데이터가 완전히 신규임에도 성능 기여 없음
