# 발표 자료 구성안 — 풍력 발전기 스마트 예방 정비 시스템

> 발표 시간: 약 10~15분 기준 | 루브릭 8개 항목 커버

---

## 슬라이드 구성 (총 12장 권장)

---

### 슬라이드 1 — 표지

**제목**: 풍력 발전기 스마트 예방 정비 시스템
**부제**: YOLOv8 기반 드론 영상 결함 탐지 + 심각도 알림

---

### 슬라이드 2 — 배경 & 문제 정의 (1분)

**핵심 메시지**: "왜 이게 필요한가"

- 풍력 터빈의 특성: 해상·산간 → 접근 어려움 → 사람 점검 비효율
- 문제: 사전에 손상을 못 잡으면 → 블레이드 파손 → 수백억 손실
- 해결: **드론 + AI → 실시간 탐지 → 즉시 알람**

**핵심 이미지**: 터빈 드론 촬영 사진 + 손상 bbox 예시

---

### 슬라이드 3 — 데이터 EDA (2분)

**핵심 메시지**: "데이터를 제대로 이해했다"

**3-1. 기본 현황**

| 항목 | 수치 |
|---|---|
| 전체 이미지 | 13,470장 |
| 정상 이미지 | 10,475장 (77.8%) |
| 결함 이미지 | 2,995장 (22.2%) |

**3-2. 핵심 발견 (3가지)**

```
① 클래스 불균형: Dirt(581) : Damage(8,770) = 1:15
② 소형 객체: Damage 86%가 64×64픽셀 이하
③ 정상 과다: 그대로 학습 시 배경 희석 → Precision 붕괴
```

**사용 차트**: 클래스 분포 막대 / bbox 크기 히스토그램

---

### 슬라이드 4 — 데이터 전처리 (1분)

**핵심 메시지**: "문제를 데이터 설계로 해결했다"

| 문제 | 대응 전략 |
|---|---|
| 정상 이미지 과다 (77.8%) | 빈 라벨 background image로 전체 활용 → FP 억제 학습 |
| Dirt 극소수 (1:15 불균형) | ×8 증강 실험 (581 → 4,648개) |
| Damage 다양성 부족 | 외부 데이터 편입 실험 (blade v7i, crack→Damage 리맵핑) |

**외부 데이터 신뢰성**: 3종 검토 → 2종 중복 확인(미사용), 1종만 실험 편입
**결과 스포일러**: 증강·외부 데이터 모두 비교 실험에서 역효과 확인 → 원본 full dataset 채택 (슬라이드 8)

---

### 슬라이드 5 — 모델 선정 (1분)

**핵심 메시지**: "왜 YOLO인가 + 왜 v8s인가"

**YOLO 선택 이유 (한 줄)**
> 단일 forward pass 실시간 탐지 + 소형 객체 multi-scale head + COCO 사전학습

**4종 비교 결과**

| 모델 | mAP50 | Recall |
|---|---|---|
| YOLOv8n | 0.794 | 0.747 |
| **YOLOv8s ← 선정** | **0.818** | 0.750 |
| YOLO11n | 0.788 | 0.741 |
| YOLO11s | 0.799 | 0.746 |

→ YOLOv8s: 최고 정확도 + 드론 탑재 가능한 경량 (22MB)

---

### 슬라이드 6 — 평가 지표 설계 (1분)

**핵심 메시지**: "왜 Recall을 1순위로 잡았는가"

```
FN (미탐) = 손상을 놓침 = 블레이드 파손 → 사고 직결
FP (오탐) = 없는데 있다고 탐지 → 재확인으로 보완 가능
```

| 순위 | 지표 | 이유 |
|---|---|---|
| 1 | Recall | FN = 사고 직결 |
| 2 | Damage AP50 | 서비스 가중치 Damage=10 (Dirt의 5배) |
| 3 | mAP50 | 참고 지표 (Dirt/Damage 동등 평균이라 왜곡) |

---

### 슬라이드 7 — 실험 과정 (2분)

**핵심 메시지**: "가설→실험→분석의 체계적 접근"

**실험 흐름도**

```
베이스라인 (full dataset, ep50)
  → 문제 진단: 클래스 불균형 (1:15), Damage 다양성 부족
    ↓
  [가설 1] Dirt×8 증강 → 불균형 완화?
  결과: Dirt에 치우친 학습 → Recall·Damage AP50 하락
    ↓
  [가설 2] 외부 Damage 데이터(blade) → 다양성 향상?
  결과: 도메인 불일치(384×384, 다른 터빈) → 기여 없음
    ↓
  [가설 3] 증강 + blade 조합 → 시너지?
  결과: 두 변화가 노이즈로 작용 → 전 지표 최저
    ↓
  [가설 4] conf 하향(0.25→0.20) → 소형 객체 탐지 포함?
  결과: mAP50 0.7016→0.7137 개선 — 실제 TP 존재 확인
    ↓
  결론: 베이스라인(full dataset) + conf=0.20 최종
```

**총 10개 실험** (모델 4 + 데이터셋 4 + conf 2)

### 슬라이드 8 — 최종 결과 비교 (1분)

### 슬라이드 9 — 최종 모델 & conf 최적화 (1분)

**최종 선정**
```
YOLOv8s | full dataset | 50 epoch
best.pt + conf=0.20
```

**conf=0.20 선정 이유**
- 소형 객체(64×64 이하)는 낮은 confidence로 탐지됨
- conf 0.20 적용 시 mAP50 0.7016 → **0.7137** 개선 (TP 증가 확인)

**최종 성능 (목표 달성)**

| 지표 | 목표 | 달성 |
|---|---|---|
| Recall | 0.70+ | **0.769 ✅** |
| Damage AP50 | 0.58+ | **0.688 ✅** |

### 슬라이드 10 — 서비스 설계 (2분)

**핵심 메시지**: "탐지를 넘어 실제 서비스로"

**심각도 스코어링 공식**

```
위험도 = Σ(클래스가중치 × 크기가중치 × confidence)
  - Damage = 10점, Dirt = 2점
  - 박스 클수록 가중치↑
```

**등급별 에스컬레이션**

| 등급 | 알림 방식 | 왜 이렇게? |
|---|---|---|
| 🟢 정상 | 로그만 | — |
| 🟡 관찰 | 대시보드 | 담당자 자율 확인 |
| 🟠 주의 | + 이메일 | 팀 공유 |
| 🔴 위험 | + Slack 즉시 경보 | 즉각 대응 |

**오탐 관리**: conf 0.4~0.7 구간 → "재확인 필요" 플래그 (human-in-the-loop)

## Q&A 예상 질문 & 답변

**Q. Recall이 높은데 Precision이 낮으면 실제로 쓸 수 있나요?**
> A. 오탐은 재확인 플래그(confidence 0.4~0.7 구간)로 운영 레벨에서 보완합니다.
> 미탐 1건이 설비 사고로 이어지는 비용이 오탐 수십 건의 재확인 비용보다 훨씬 크기 때문에 Recall 우선 전략을 유지했습니다.

**Q. 왜 더 큰 모델(YOLOv8m/l)을 쓰지 않았나요?**
> A. 드론 온디바이스 탑재를 목표로 했기 때문입니다. YOLOv8s(22MB)는 엣지 환경에서도 동작 가능한 범위이고, 4종 비교에서 이미 최고 성능을 기록했습니다.

**Q. conf=0.20이 너무 낮지 않나요? 오탐이 많아지지 않나요?**
> A. conf=0.20은 inference 파라미터로, 0.4 미만은 서비스 레벨에서 '무시' 처리됩니다.
> 0.20~0.40 구간의 탐지는 "재확인 필요" 플래그로 분류되어 자동 경보를 내지 않습니다.

**Q. 외부 데이터(blade)를 추가했는데 왜 성능이 오히려 떨어졌나요?**
> A. blade 데이터는 384×384 해상도의 다른 터빈 기종 이미지로, NordTank(586×371)와 도메인이 달라 오히려 노이즈로 작용했습니다. 동일 도메인 데이터 확보가 더 효과적입니다.
