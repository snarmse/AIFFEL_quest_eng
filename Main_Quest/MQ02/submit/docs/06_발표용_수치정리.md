## 2. 데이터셋 구성 5종

| 데이터셋 | Train | Val | Labeled | Background | 사용 목적 |
|---|---|---|---|---|---|
| labeled_only | 2,396 | 599 | 2,396 | 0 | 모델 선정 실험 (val 기준 다름) |
| **full (baseline)** | **10,776** | **2,694** | **2,394** | **8,382** | **최종 채택** |
| full + Dirt×8 | 14,400 | 2,694 | 6,018 | 8,382 | 비교 실험 |
| full + blade | 12,250 | 2,694 | 3,853 | 8,397 | 비교 실험 |
| full + Dirt×8 + blade | 15,874 | 2,694 | 7,477 | 8,397 | 비교 실험 |

**왜 full dataset을 채택했나?**
- 정상 이미지를 빈 라벨(background image)로 학습 → YOLO가 배경을 배경으로 인식하도록 학습
- labeled_only 학습 시 정상 이미지에서 FP 폭증 (배경을 한 번도 본 적 없음)
- full dataset이 Precision 안정화에 직결

## 3. 평가 지표 설계

### 지표 우선순위

| 순위 | 지표 | 이유 |
|---|---|---|
| **1** | **Recall** | FN = 결함 미탐지 = 사고 직결. FP는 재확인으로 보완 가능 |
| **2** | **Damage AP50** | 서비스 가중치 Damage=10 (Dirt의 5배). 서비스 가치와 직결 |
| 참고 | mAP50 | Dirt=Damage 동등 평균 → 클래스 불균형 상황에서 왜곡 |

### 목표값

| 지표 | 목표 | 근거 |
|---|---|---|
| Recall | 0.70+ | 결함 30% 이상 미탐지 시 서비스 신뢰 붕괴 |
| Damage AP50 | 0.58+ | Damage 탐지율 일정 이상 보장 필요 |

### FN vs FP 비용 비교

```
FN (미탐지):  결함 → 정상 판정
              → 블레이드 파손 방치
              → 수억원 손실 / 안전 사고
              → 비용 복구 불가

FP (오탐지):  정상 → 결함 판정
              → 재확인 작업 필요
              → 인력·시간 비용 발생
              → human-in-the-loop로 관리 가능

결론: FP 비용 << FN 비용  →  Recall 극대화 전략 채택
```

## 4. 모델 선정 실험 (Phase 1)

**조건**: labeled_only 데이터셋, 20 epoch, labeled val (599장) 기준

| 모델 | mAP50 | mAP50-95 | Recall | Precision | 파라미터 |
|---|---|---|---|---|---|
| YOLOv8n | 0.794 | 0.553 | 0.747 | 0.818 | 3.2M / 6MB |
| **YOLOv8s** ✅ | **0.818** | **0.575** | **0.750** | **0.854** | **11M / 22MB** |
| YOLO11n | 0.788 | 0.534 | 0.741 | 0.812 | 2.6M / 5.4MB |
| YOLO11s | 0.799 | 0.559 | 0.746 | 0.851 | 9.4M / 18MB |

**선정: YOLOv8s**
- mAP50 최고 (0.818) + Precision 최고 (0.854)
- 22MB — 드론 탑재 가능한 경량 경계점
- YOLO11s 대비 mAP50(+0.019), Recall(+0.004) 소폭 우세

## 5. 데이터셋 비교 실험 (Phase 2)

**조건**: YOLOv8s, 50 epoch, 동일 val 2,694장 기준 (4종 공통)

| 데이터셋 | Precision | Recall | mAP50 | Damage AP50 | 결론 |
|---|---|---|---|---|---|
| **Baseline (full)** ✅ | **0.710** | **0.769** | **0.752** | **0.688** | 전 지표 최고/동등 |
| + Dirt×8 | 0.674 | 0.751 | 0.718 | 0.684 | Damage 학습 신호 희석 |
| + Blade | 0.710 | 0.763 | 0.748 | 0.688 | 도메인 불일치, 기여 없음 |
| + Dirt×8 + Blade | 0.645 | 0.735 | 0.726 | 0.682 | 노이즈 누적, 전 지표 최저 |

**왜 Dirt×8이 역효과였나?**
- Dirt 증강 (468 → 4,648개): Damage 대비 비율 개선 시도
- 결과: 학습 배치에서 Dirt가 과도하게 등장 → Damage 탐지 신호 희석
- Recall 하락 + Damage AP50 하락

**왜 blade 외부 데이터가 역효과였나?**
- blade (Roboflow): 384×384px, 다른 터빈 기종 (다른 형태의 결함 패턴)
- NordTank: 586×371px, 수평축 터빈
- 도메인 불일치 → 학습 노이즈로 작용

## 6. conf 최적화 (Phase 3)

**val() vs predict() 동작 차이**

| 함수 | conf 반영 | Recall 변화 | 용도 |
|---|---|---|---|
| `val()` | 내부 임계값 스윕 | 변화 없음 | 평가 지표 산출 |
| `predict()` | conf 파라미터 적용 | conf 낮출수록 증가 | 실제 추론 |

> mAP50은 val()로 계산하므로 conf를 낮춰도 실제 TP가 늘어나면 mAP50이 올라감

**conf 비교 결과**

| conf | mAP50 | 변화 |
|---|---|---|
| 0.25 (기본값) | 0.7016 | — |
| **0.20** ✅ | **0.7137** | **+0.0121** |

- 소형 객체(19×20px 중앙값)는 낮은 confidence로 탐지됨
- conf 0.20 적용 시 실제 TP 증가 확인 → mAP50 개선
